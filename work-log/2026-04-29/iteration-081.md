4. Frontend: matrix coverage. The new `PoLineNegotiationTable` and `PoLineNegotiationRow` must render correctly across 5 LineItemStatus values (PENDING, MODIFIED_BY_VENDOR, MODIFIED_BY_SM, ACCEPTED, REMOVED) x 2 roles (VENDOR, SM) x 3 round_count values (0, 1, 2). The action visibility rules: Accept appears only when the counterparty was last to move; Modify and Remove appear on PENDING / MODIFIED_BY_* states regardless of last actor; Force Accept and Force Remove appear only when role is SM, round_count is 2, and status is MODIFIED_BY_*; ACCEPTED and REMOVED rows render no actions. These rules ship verbatim from the iter 057 component and are tested in the new spec.
5. Frontend: testid contract. Each row exposes `po-line-{part_number}` (root), `po-line-status-{part_number}` (pill), `po-line-actions-{part_number}` (action group), `po-line-action-{accept|modify|remove|force-accept|force-remove}-{part_number}` (each action button), `po-line-error-{part_number}` (server error slot), `po-line-details-toggle-{part_number}` (View changes toggle), `po-line-diff-{part_number}` and `po-line-history-{part_number}` (revealed children), and `po-line-history-toggle-{part_number}` / `po-line-history-list-{part_number}` (history sub-toggle). The modify modal exposes `po-line-modify-modal-{part_number}` plus `po-line-modify-{quantity|unit-price|uom|description|hs-code|country|rdd}` per field and `po-line-modify-{cancel|submit}` per button. The force confirm overlays expose `po-line-force-{accept|remove}-confirm-{part_number}` and `po-line-action-force-{accept|remove}-confirm-{part_number}`. The submit bar exposes `po-submit-response-bar`, `po-submit-response-round`, `po-submit-response-hint`, `po-submit-response-btn`, `po-submit-response-confirm`, `po-submit-response-confirm-btn`, `po-submit-response-delta-{accepted|modified|removed}`, and `po-submit-response-error`.
6. Frontend: primitive accessible-name convention. Per [CLAUDE.md](CLAUDE.md) "Primitive accessible-name convention", the new components map onto the convention as follows. `PoLineNegotiationTable` is a list container and accepts a `label?: string` prop forwarded to `aria-label` on the section root (default "Line items under negotiation"). `PoLineModifyModal` is a titled container; the visible "Modify line {part_number}" h2 owns a `crypto.randomUUID()` id wired to the dialog's `aria-labelledby`. `PoSubmitResponseBar` is a titled region; the round label h-element owns the same generated id pattern. `PoLineNegotiationRow` is a per-entity card (no title element) and exposes structural state via `data-status` plus the StatusPill primitive; no extra plumbing. `PoLineDiff` and `PoLineEditHistoryTimeline` are list containers; they render `<dl>` / `<ol>` / `<ul>` whose accessible name comes from the surrounding row, so no `label` prop is needed.
7. Frontend: modify modal validation. The new modal rejects empty / whitespace-only strings on UoM, Description, HS code, and Country of Origin (per CLAUDE.md "name the invalid inputs"). Quantity must be a non-negative integer. Unit price must be > 0 unless quantity is 0 (in which case the line will route to REMOVED server-side; the modal surfaces a hint). HS code must match `^[0-9.]{4,}$`. The modal computes a changed-fields delta against `line.*` and silently cancels if no field changed.
8. Frontend: status pill mapping. `PoLineNegotiationRow` uses [`StatusPill`](frontend/src/lib/ui/StatusPill.svelte) with the per-status tone and label encoded in a `Readonly<Record<LineItemStatus, {tone, label}>>` constant: PENDING -> gray "Pending", MODIFIED_BY_VENDOR -> blue "Modified by vendor", MODIFIED_BY_SM -> orange "Modified by SM", ACCEPTED -> green "Accepted", REMOVED -> red "Removed", REJECTED -> red "Rejected" (legacy fixtures only).
9. Frontend: mobile (390px) layout. `PoLineNegotiationRow` collapses the head row to vertical stacking and gives every action button a 50% min-width so two actions sit per row. `PoSubmitResponseBar` adds `env(safe-area-inset-bottom)` to its bottom padding so the iOS home indicator does not overlap the Submit button. Diff and history rows collapse to single-column at 640px and below.
10. DDD vocab: assess whether "Hand-Off", "Force Override", "Convergence", "Editable Line Fields", "Line Edit History", and "Negotiation Round" appear in [docs/ddd-vocab.md](docs/ddd-vocab.md) and propose additions before iteration close per CLAUDE.md rule 6.1.
