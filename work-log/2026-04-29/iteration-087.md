# Iteration 087 — `/invoice/[id]` detail revamp (Phase 4.3 Tier 2)

## Context

Iter 086 ported `/invoices` (list) under `(nexus)` on Phase 4.0 primitives. The detail page at [frontend/src/routes/invoice/[id]/+page.svelte](frontend/src/routes/invoice/[id]/+page.svelte) (~204 LOC) still imports three legacy components: [`StatusPill`](frontend/src/lib/components/StatusPill.svelte), [`DisputeDialog`](frontend/src/lib/components/DisputeDialog.svelte), [`ActivityTimeline`](frontend/src/lib/components/ActivityTimeline.svelte). It renders a flat sequence of class-based `.section.card` blocks with a bottom `.actions` button row gated by inline status checks. Iters 077 and 083 supply the precedent: move the route under `(nexus)`, replace the header with a `DetailHeader`-backed component, surface a single role-aware action rail (inline at desktop, sticky-bottom at mobile), and wrap sections in `PanelCard`s. The shared event-label dictionary at [frontend/src/lib/po/event-labels.ts](frontend/src/lib/po/event-labels.ts) already includes every `INVOICE_*` event; promoting it to `$lib/event-labels.ts` is the cleanup once invoice detail becomes the second consumer. [`PoInvoicesPanel`](frontend/src/lib/po/PoInvoicesPanel.svelte) is the last consumer of the legacy `StatusPill` and migrates here so the legacy component can retire.

## JTBD

When I open `/invoice/[id]` as SM/ADMIN, I want a `(nexus)`-shell page with a compact header (back link to `/invoices`, invoice number, PO number + created-date subtitle, status pill), a single role-aware action rail at top-right that collapses into a sticky-bottom bar on mobile, panels for metadata + line items + activity, and a dispute-reason panel rendered only when the invoice is DISPUTED, so I can act on the invoice without scrolling through a five-card pre-revamp layout. As VENDOR I want Submit on DRAFT and Download PDF on every status. As SM I want Approve / Dispute on SUBMITTED, Pay on APPROVED, Resolve on DISPUTED, and Download PDF on every status. As PROCUREMENT_MANAGER, FREIGHT_MANAGER, or QUALITY_LAB I want Download PDF only.

## Notes

The retrofit followed the iter-077 + iter-083 PO precedent line-for-line: route move under `(nexus)`, [`InvoiceDetailHeader`](frontend/src/lib/invoice/InvoiceDetailHeader.svelte) wrapping `DetailHeader`, [`InvoiceActionRail`](frontend/src/lib/invoice/InvoiceActionRail.svelte) with desktop-inline + mobile-sticky modes, and `PanelCard`-wrapped sub-sections. The action rail diverged from `PoActionRail` only in shape: `SUBMITTED` simultaneously surfaces Approve (primary) + Dispute (secondary) since both are valid SM actions on the same status, whereas every PO status has at most one primary action per role. The dispute reason flow stayed in a dedicated [`InvoiceDisputeModal`](frontend/src/lib/invoice/InvoiceDisputeModal.svelte) (not a separate page) because the textarea is single-field and roundtrip is one POST. PROCUREMENT_MANAGER + FREIGHT_MANAGER see Download PDF only; QUALITY_LAB never reaches `/invoice/:id` because `canViewInvoices` excludes them — the iter doc had originally listed QL alongside PM/FM in the JTBD and tests, corrected during test green-up.

`window.open` for PDF download made the PDF test brittle on first attempt — `popup.url()` returned `':'` because Playwright surfaces popups before they navigate. Switched to a `page.addInitScript`-based `window.open` shim that records the requested URL into `window.__openedUrls`, which captures the call synchronously without depending on popup load timing. Same pattern applies to PO detail's Download PDF if its tests ever break.

The shared event-label dictionary moved out of `frontend/src/lib/po/event-labels.ts` to [frontend/src/lib/event-labels.ts](frontend/src/lib/event-labels.ts) so both `PoActivityPanel` and `InvoiceActivityPanel` consume one canonical module. The original PO-only home was a comment-acknowledged temporary parking spot. [`PoInvoicesPanel`](frontend/src/lib/po/PoInvoicesPanel.svelte) migrated from the legacy [`$lib/components/StatusPill.svelte`](frontend/src/lib/components/StatusPill.svelte) to the Phase 4.0 [`$lib/ui/StatusPill.svelte`](frontend/src/lib/ui/StatusPill.svelte) primitive in the same iter, which freed the three legacy components — `StatusPill`, `DisputeDialog`, `ActivityTimeline` — for full deletion. Three `activity-timeline.spec.ts` invoice-detail specs targeting the legacy `.timeline-entry` / `.timeline-dot` / `.timeline` markup migrated to the new `invoice-activity-feed` testid + `.dot.orange` tone-class assertion. The spec file name remains `activity-timeline.spec.ts` even though the legacy timeline retired — renaming would just churn imports without changing the assertions; revisit if a future iter touches the file.

Phase 4.3 is now feature-complete on the read surfaces: `/invoices` list (iter 086) + `/invoice/:id` detail (this iter). The remaining invoice surface is the Create flow, currently entered from `/po/:id` via the `CreateInvoiceDialog` component on PO detail, which itself is unmigrated from the legacy `frontend/src/lib/components/` directory and still uses native `<input class="input">` markup. That's the natural Phase 4.3 Tier 3 scope. 696 backend + 300 Playwright (282 → 300, +18) all green.
